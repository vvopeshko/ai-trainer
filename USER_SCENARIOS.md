# USER_SCENARIOS — AI Trainer

Юзер-сценарии мини-аппа (и связанных с ним сценариев бота) — что делает пользователь, в какой момент и через какие экраны. Документ служит основой для прототипирования и проверки продуктовых гипотез.

Продуктовый контекст — в [BRD.md](BRD.md). Технические решения — в [ARCHITECTURE.md](ARCHITECTURE.md). Приоритеты разработки — в [NEXT_PLANS.md](NEXT_PLANS.md).

## 1. Общие принципы

### Дихотомия бот ↔ мини-апп

| Бот (чат) — "говорим" | Мини-апп — "действуем/смотрим" |
|---|---|
| **Онбординг** — анкета через диалог (цель, опыт, ограничения, оборудование) | Структурированное логирование тренировки (подходы, веса) |
| **Создание/пересборка программы** через диалог ("сгенерь программу на силу") | Просмотр текущей программы и её упражнений |
| Задать вопрос AI-тренеру ("как правильно присед?") | Запуск следующей тренировки из программы |
| Отправить фото тренажёра → быстрый ответ "это такой-то, упражнения такие-то" | Просмотр прогресса (графики, карта мышц, история) |
| **Сводка после каждой тренировки** (тренировка + месяц) | Библиотека упражнений с видео и описанием |
| Еженедельная и ежемесячная сводки, пост-анализ плана | Настройки профиля (редактирование анкеты позже) |
| Мотивационные сообщения, напоминания | — |

**Правило:** бот — для *диалога и моментальных* ответов (вопросы, генерация программы, сводки, поощрение). Мини-апп — для *структурных действий*, требующих фокуса или визуала (логирование подходов, просмотр программы и прогресса).

**В MVP онбординг и редактирование анкеты не делаем в мини-аппе** — возвращаемся к этому в отдельной итерации, если окажется, что диалога недостаточно.

### Точки входа в мини-апп

1. **Menu Button** в Telegram-боте (всегда внизу чата)
2. **Inline-кнопка `web_app`** в сообщениях бота (контекстные, например "открыть эту тренировку")
3. **Прямая ссылка** `/` → главный экран
4. **Deep-link с параметром** — например, `/?startapp=workout_abc123` сразу открывает конкретную тренировку
5. **Из других сценариев**: push-уведомление → tap → мини-апп с нужным экраном

### Принципы UX

- **Mobile-first.** Всё работает одной рукой, тапы 44px+, критичные действия в нижней части экрана.
- **Работает в зале.** Между подходами 60–90 секунд — не нарезаем процесс паузами, лоадерами, модалками. Оптимистичные апдейты, критичные действия в один тап.
- **Haptic feedback** на завершении действий (подход засчитан, отдых начался).
- **Минимум ввода.** Автоподстановка из прошлой тренировки. Текстовые поля — только там, где иначе нельзя.
- **Офлайн-дружелюбно.** Если интернет пропал между подходами — ввод сохраняется локально, синхронизируется при возврате.
- **Тёмная тема** с первого дня (см. ARCHITECTURE.md §5.6).

### Персоны

- **Новичок** (0–6 мес. в зале) — нужны подсказки, чёткая программа, "делай это сейчас".
- **Любитель** (средний уровень) — нужна структура, аналитика, возможность отходить от плана.

## 2. MVP-сценарии (итерации 1–5)

### 2.1 🟢 Онбординг и первая программа (в боте)

**Триггер:** пользователь впервые нашёл бота и нажал Start.

**Цель:** за 5–7 минут получить персональную программу и понять, как с ней работать. Весь процесс проходит в диалоге с ботом — без мини-аппа.

**История:**
Новичок Саша открывает бота. Приветственное сообщение: "Привет! Я AI Trainer. За пару минут соберу тебе программу тренировок под твою цель. Готов?". Бот задаёт вопросы один за другим, каждый с inline-клавиатурой для быстрых ответов:

1. "Какая цель?" → [Похудение / Масса / Сила / Тонус]
2. "Сколько опыта в зале?" → [Новичок / Любитель / Продвинутый]
3. "Пол?" → [М / Ж]
4. "Возраст?" → текстовый ответ ("25")
5. "Рост / вес?" → "180 / 75"
6. "Есть ли травмы или ограничения?" → [Нет / Поясница / Колени / Плечи / Написать свои]
7. "Сколько раз в неделю готов тренироваться?" → [2 / 3 / 4 / 5+]
8. "Какое оборудование доступно?" → [Обычный коммерческий зал / Только гантели дома / Только штанга / Написать]

Где это уместно, даём юзеру писать ответ своими словами — бот парсит через LLM.

После последнего вопроса — "Секунду, подбираю программу...". Через ~10 сек бот присылает:
> "Готово! Твоя программа: **Split A/B** на 4 недели, 3 тренировки в неделю.
> Тренировка A — грудь + трицепс + плечи (5 упражнений)
> Тренировка B — спина + бицепс + ноги (6 упражнений)
> 
> [🏋️ Открыть программу] [❓ Что означают эти упражнения?]"

Тап на "Открыть программу" → мини-апп открывается на экране просмотра программы (см. 2.2).

**Ключевые экраны:**
- Всё в боте — отдельных экранов в мини-аппе нет.
- Финальное сообщение с inline-кнопкой на мини-апп для просмотра программы.

**Handoff:** **Бот → мини-апп.** После завершения онбординга бот даёт кнопку, открывающую мини-апп на экране "Моя программа".

**Edge cases:**
- Юзер бросил анкету посредине → бот сохраняет состояние, при возврате спрашивает "Продолжим с того места, где остановились?"
- Юзер ответил не по шаблону ("я так себе в зале", "120 кг / 180 см") → LLM парсит в структуру. Если не получилось — переспросим.
- Нет ответа от LLM на генерации программы → "Попробую ещё раз через минуту" + fallback-шаблон (жёстко прошитая программа под цель/уровень).
- Травма → AI учитывает в генерации (например, убирает становую при пояснице).

**Метрика успеха:** % юзеров, прошедших онбординг до получения программы (completion rate anket'ы).

### 2.2 🟢 Просмотр программы и запуск тренировки

**Триггер:** юзер открывает мини-апп (через Menu Button, inline-кнопку из бота или просто в любой момент).

**Цель:** увидеть свою программу, понять, какая тренировка следующая, и начать её.

**Ключевое:** мы **не привязываем тренировки к дням недели**. Программа — это последовательность тренировок (A, B, A, B, ...), которую юзер проходит в своём темпе. Хоть каждый день, хоть раз в неделю. Анализ "насколько реально соответствует плану" делается пост-фактум (см. 3.3).

**История:**
Любитель Вика открывает мини-апп. Главный экран показывает:

- Шапку "Моя программа — Split A/B (неделя 2)"
- Прогресс по циклу: "Сделано 5 тренировок из 12 запланированных на месяц"
- Карточку **"Следующая: Тренировка B — спина, бицепс"** (6 упражнений, ~50 мин) + большая кнопка "Начать"
- Под ней — "Сделать другую вместо этой": переключиться на A, B или внеплановую тренировку (например, мобилити)
- Список последних сделанных тренировок: "Вчера: A • 2 дня назад: B • 4 дня назад: A"

Вика жмёт "Начать" → режим тренировки (см. 2.3).

**Ключевые экраны:**
- Главный экран мини-аппа: "Моя программа" с прогрессом цикла и карточкой следующей тренировки
- Превью тренировки (упражнения списком, примерная длительность, подходы/повторы)
- Режим тренировки (см. 2.3)

**Handoff:** нет прямого handoff — сценарий внутри мини-аппа. Но главный экран открывается в том числе и по deep-link из бота (кнопки "Открыть программу", "Начать тренировку").

**Edge cases:**
- Юзер только что завершил тренировку A → следующая автоматически = B (чередование).
- Юзер хочет сделать A два раза подряд (работает над приоритетной группой) → кнопка "Сделать другую" позволяет выбрать. В базу записывается фактический порядок.
- Юзер давно не тренировался (7+ дней) → бот **отдельно** (не мини-апп!) шлёт ободряющее сообщение "Готов вернуться?". Если юзер возвращается — программа продолжается с того места, где остановился.
- Юзер в середине цикла хочет поменять программу → кнопка "Сменить программу" → handoff в бот, чат с генерацией новой (см. 2.7).

**Метрика успеха:** % сессий в мини-аппе, в которых юзер нажал "Начать тренировку" (conversion открытия в запуск).

### 2.3 🟢 Логирование подхода (ядро)

**Триггер:** юзер выполняет упражнение в зале между подходами.

**Цель:** быстро зафиксировать вес × повторения с минимумом тапов. Это **главный экран в жизни пользователя** — в него смотрят десятки раз за тренировку.

**История:**
Саша встал у штанги, открыл мини-апп (или вернулся в него — он оставался открытым). На экране текущее упражнение "Жим лёжа", прошлая тренировка: `60×10, 60×10, 60×8`. Автоподстановка предлагает те же параметры. Саша выполнил первый подход `62.5×10`, тапнул "Готово", цифры пополнились. Таймер отдыха 90с запустился автоматически. По истечении таймера — хаптик + тихий звук, возврат к экрану подхода. Через 3 подхода — предложение "Следующее упражнение" с плавным переходом.

**Ключевые экраны:**
- Режим тренировки — упражнение с превью техники (короткий gif или статик), 3 подхода предзаполненные, таймер отдыха, кнопка "Следующее упражнение" и "Завершить тренировку" (сверху, всегда доступна).
- Карточка подхода: вес (numeric stepper +/- 2.5 кг), повторы (+/-1), "Готово". RPE — **только для intermediate/advanced** (определяется по `UserProfile.experienceLevel`). Beginner'ы RPE не видят; могут включить через настройки.
- Таймер отдыха: автозапуск после "Готово", полноэкранная плашка со секундомером обратного отсчёта + кнопка "Пропустить" (на виду). Время отдыха берётся из упражнения (compound = 120–180с, isolation = 60–90с).
- Мини-карта тренировки (горизонтальные точки — выполнено/в процессе/дальше). Тап на точку — переход к упражнению (гибкий порядок).

**Принципы режима тренировки:**
- **Гибкий порядок.** Юзер может тапнуть на любое упражнение из списка и перейти к нему без строгого следования плану. Прогрессом считается "все залогированы", а не "в правильной последовательности".
- **Добавление внепланового упражнения.** Кнопка "+ упражнение" → поиск по библиотеке → добавить в текущую тренировку. Отмечается как "внеплановое".
- **Пропуск упражнения.** В меню упражнения ("⋯") → "Пропустить". Фиксируется в БД.
- **Все отклонения логируются** и попадают в ежемесячный пост-анализ (3.3) — AI-тренер может предложить скорректировать программу, если видит устойчивый паттерн.

**Handoff:**
- Юзер нажал "Задать вопрос тренеру" в экране упражнения → переход в бот с предзаполненным контекстом ("Я сейчас делаю Жим лёжа, 60 кг, ...").
- Юзер нажал "Сфотографировать тренажёр" → временно закрывает мини-апп, открывает камеру Telegram → фото улетает боту → бот возвращает результат.

**Edge cases:**
- Юзер добавляет лишний подход сверх плана ("ещё один") → одна кнопка "+ подход".
- Юзер хочет заменить упражнение (занято/больно) → "Заменить" → AI в чате предлагает 2–3 замены, выбор возвращает в тот же экран.
- Юзер потерял интернет между подходами → optimistic UI + локальная очередь pending-подходов в localStorage. Плашка "Синхронизируется, когда вернётся связь". Лимит очереди — 50 записей.
- Юзер закрыл мини-апп посреди тренировки → при возврате "Продолжить тренировку?" (yes/no). Pending-подходы из очереди подтягиваются автоматически.

**Метрика успеха:** среднее число залогированных подходов на тренировку; % тренировок, завершённых до конца (не брошенных посередине).

### 2.4 🟢 Завершение тренировки — сводка от бота

**Триггер:** юзер в мини-аппе нажал "Завершить тренировку" (или автоматически, если залогировал все подходы из плана).

**Цель:** зафиксировать результат, получить осмысленный фидбек и связку "эта тренировка → в контексте месяца".

**История:**
Саша закончил последнее упражнение. В мини-аппе короткий экран "Готово!" с цифрами (тоннаж, время, рекорды если есть) и опциональным вопросом "Как прошла?" (1–5 звёзд + заметка). Через пару секунд **бот в чате** присылает развёрнутое сообщение:

> 🎉 **Тренировка A завершена!**
>
> *Сегодня:* 6 подходов × среднее 10 повторов, тоннаж **6 840 кг**, время 58 мин.
> Прогресс: жим +2.5 кг (72.5→75), спина стабильно.
>
> *За месяц:* 11 тренировок из 12 запланированных (92% к плану). Тоннаж растёт +8% к прошлому месяцу. Основная нагрузка — грудь (32%) и спина (28%), ноги отстают (15%).
>
> *Наблюдение тренера:* если пропускаешь ноги — лучше сделать отдельный день, чем разбрасывать внутри A/B.
>
> [📊 Посмотреть детали] [💬 Обсудить с тренером]

**Ключевые экраны:**
- Мини-апп: короткий экран-подтверждение (тоннаж, время, рекорды, опциональный ввод "Как прошла?") — секунды на экране, дальше бот.
- Бот: развёрнутая сводка с двумя уровнями — (1) эта тренировка, (2) контекст месяца (progress, план vs факт, наблюдения, рекомендации от AI).

**Handoff:** **Мини-апп → бот.** После нажатия "Завершить" мини-апп коротко подтверждает, бот присылает развёрнутое сообщение. Кнопка "Посмотреть детали" в боте ведёт обратно в мини-апп на страницу прогресса.

**Edge cases:**
- Юзер залогировал не все подходы, но нажал "Завершить" → в сообщении бота отмечаем "Выполнено 4 из 6 упражнений. Продолжим в следующий раз?".
- Юзер закрыл мини-апп посреди тренировки и не вернулся → бот через несколько часов спрашивает "Закончил тренировку? Залогировать как завершённую?".
- Новый персональный рекорд → особо выделяем в сообщении бота ("🏆 Новый рекорд в жиме: 75 кг × 6!").
- Первая тренировка (первый раз в жизни) → бот добавляет "Поздравляю с первой тренировкой! Продолжай, дальше будет легче."

**Метрика успеха:** % тренировок с явным "Завершить" (а не брошенных); открываемость ссылки "Посмотреть детали" в сообщении бота.

### 2.5 🟢 Распознавание тренажёра по фото ⭐ (итерация 4)

**Триггер:** юзер в зале стоит у непонятного тренажёра.

**Цель:** за 10–15 секунд понять, что это за тренажёр и какое упражнение на нём делать.

**История:**
Вика увидела в зале странный тренажёр, который раньше не трогала. Открыла бота в Telegram (не мини-апп — здесь бот быстрее), сфотографировала тренажёр, отправила в чат. Через ~10 секунд получила: "Это вертикальный блок (тяга верхнего блока широким хватом). Основные мышцы: широчайшие, бицепс. Рекомендую 3×12 с весом 70% от вашего обычного на гребной. [🎥 Видео техники] [📝 Открыть в тренировке]".

**Ключевые экраны:**
- Весь сценарий в боте — отдельного экрана в мини-аппе нет.
- При тапе на "Открыть в тренировке" → мини-апп открывается сразу на экране логирования этого упражнения с deep-link.

**Handoff:** **Bot → мини-апп.** Ключевая точка стыковки. Бот передаёт в мини-апп параметр `?startapp=identified_bench_press_at_2026-04-22_17-15` → мини-апп парсит и открывает логирование с предзаполненным упражнением.

**Edge cases:**
- LLM не уверен (`confidence < 0.6`) → бот пишет "Не уверен, это похоже на X или Y, уточни пожалуйста". Юзер выбирает из вариантов или пишет своими словами.
- LLM вернул тренажёр, которого нет в нашей базе упражнений → показываем описание техники как есть (из LLM), без кнопки "открыть в тренировке" (нечего открывать).
- Юзер отправил не фото тренажёра (котик, скриншот) → "Кажется, это не тренажёр. Попробуй сфотографировать тренажёр целиком."

**Метрика успеха:** % пользователей, использовавших фичу хотя бы раз в первые 7 дней; среднее число запросов в неделю; % случаев, когда юзер подтвердил (нажал на предложенное упражнение).

---

### 2.6 🟢 AI-чат тренер в моменте

**Триггер:** юзер задал вопрос боту в любой момент (в зале, дома, перед тренировкой).

**Цель:** быстро получить осмысленный ответ с учётом своего контекста.

**История:**
Саша во время подхода почувствовал дискомфорт в пояснице. Написал боту: "поясница тянет на приседе, что делать?". Бот через ~4 секунды: "Останови подходы на приседе сегодня. Причины могут быть: плохая мобильность тазобедренных, слабый кор, неправильный хват, слишком большой вес. Рекомендую: (1) заменить присед сегодня на жим ногами 3×10, (2) после тренировки 5–10 минут мобилити поясницы, (3) если повторится — обратись к врачу. Заменить сейчас? [Да, заменить на жим ногами] [Нет, пропущу упражнение]".

**Ключевые экраны:**
- Диалог в боте. Inline-кнопки для действий ("Заменить", "Пропустить").
- Если выбрал "Заменить" → бот делает замену в активной тренировке (вызывает backend) → мини-апп при следующем открытии показывает обновлённую программу.

**Handoff:** в основном односторонний (юзер пишет в бот), но действия из бота могут модифицировать состояние мини-аппа.

**Edge cases:**
- Медицинский симптом ("чувствую боль в груди") → бот сразу отправляет к врачу, не даёт советов по тренировке.
- Юзер спрашивает вне контекста тренировок ("как дела?") → бот возвращает в тему: "Готов тренироваться?" или просто дружелюбно.
- Контекст длинный (последние 20 вопросов) → выдираем только релевантное, чтобы не разбухнуть в токенах.

**Метрика успеха:** сообщений в неделю на активного юзера; ретеншн юзеров, задавших хотя бы один вопрос (гипотеза: активные чатчики ретенятся сильно лучше).

---

### 2.7 🟢 Создание / пересборка программы через диалог

**Триггер:** юзеру надоела текущая программа, сменились цели, сменился зал, вернулся после паузы и хочет что-то другое.

**Цель:** за пару минут диалога получить новую программу с учётом изменившегося контекста.

**История:**
Саша тренируется месяц по Split A/B, но понимает, что хочется больше силы, а не массы. Пишет боту: "Хочу новую программу, на силу". Бот не спрашивает заново всю анкету — у него уже есть твой профиль. Задаёт только что-то уточняющее:

> "Окей, под силу. Пара уточнений:
> 1. Какие основные движения в приоритете? [Присед, Жим, Тяга / Все три / Выбрать]
> 2. Частота та же (3 раза в неделю) или другая?
> 3. Сохранить текущую Split A/B или попробовать новый формат (Full Body / Upper-Lower)?"

Юзер отвечает. Бот генерит. Присылает: "Готово! **Upper / Lower на 6 недель, 4 тренировки в неделю**. [Открыть программу] [Подробнее]". Открытие → мини-апп с новой активной программой.

**Также доступно свободной формой:** "хочу что-то попроще", "плечо болит, подстрой", "зал сменил, там нет штанги". Бот интерпретирует, задаёт уточняющие вопросы и генерит.

**Ключевые экраны:**
- Всё в боте, кроме итогового "открой программу" → мини-апп.
- В мини-аппе юзер видит уже новую активную программу (старая уезжает в историю).

**Handoff:** **Бот → мини-апп** после готовности. Также возможен **мини-апп → бот** через кнопку "Сменить программу" на экране программы в мини-аппе.

**Edge cases:**
- Юзер просто сказал "новую программу" без деталей → бот задаёт 2–3 минимальных уточнения (цель всё та же? частота? хочешь что-то поменять?) и генерит.
- Юзер хочет **временно** отклониться (отпуск, сессия, болезнь) → бот предлагает "лёгкий режим" на N дней, не пересобирая основную программу. По возврату — "вернулся? продолжаем?".
- Юзер просит то, что противоречит профилю (например, "пусть будет становая", но у него поясница) → бот мягко отговаривает с объяснением и предлагает альтернативу.

**Метрика успеха:** % активных юзеров, хоть раз пересобравших программу; среднее количество программ за жизненный цикл юзера.

## 3. Расширения MVP (итерация 5 и далее)

### 3.1 🟡 Просмотр прогресса

**Триггер:** юзер открывает вкладку "Прогресс".

**Цель:** увидеть, растёт ли он и в чём именно.

**История:**
Любитель Вика через неделю тренировок заходит во вкладку "Прогресс". Видит:
- Карту тела с подсвеченными мышцами (чем ярче — тем больше тоннажа за выбранный период).
- Графики по базовым движениям (жим/присед/тяга): история весов и 1RM.
- Тоннаж по тренировкам (bar chart) с трендом.
- Счётчики: тренировок за неделю/месяц, средний интервал между тренировками, текущий прогресс по активной программе ("9 из 12 тренировок цикла").

**Ключевые экраны:**
- Tabbed: "Неделя / Месяц / Всё время"
- Карта мышц (SVG), tap → детализация "Грудь: 3 упражнения, 12 подходов, 1 440 кг тоннажа"
- Графики базовых упражнений (recharts)
- Список персональных рекордов

**Handoff:** в рекорд — можно "поделиться" (отправить в бот/чат картинку с прогрессом).

**Edge cases:**
- Данных мало (< 3 тренировок) → empty state "Начни тренироваться, чтобы увидеть прогресс".
- Один тип упражнений доминирует → мягкий намёк "Мало работаешь над спиной — программа это учитывает?".

---

### 3.2 🟡 Еженедельная сводка (через бот)

**Триггер:** воскресенье вечером, node-cron в Railway. Таймзона берётся из профиля юзера.

**Цель:** отразить неделю глазами тренера: что сделано, что по плану vs факту, куда двигаться.

**История:**
Саша в воскресенье вечером получает сообщение от бота:

> 📅 **Неделя завершена**
>
> За 7 дней ты сделал **3 тренировки** (план ~3–4). Тоннаж: **18 400 кг** (+7% к прошлой неделе).
> Сделано по программе: A → B → A.
>
> 🏆 Новые рекорды: жим лёжа 72.5×8.
>
> 💡 Наблюдение: интервалы между тренировками 2–3 дня — комфортный ритм. Следующая по плану — B (спина, бицепс, ноги).
>
> [📊 Посмотреть отчёт] [▶️ Начать следующую]

Тап на "Посмотреть отчёт" → мини-апп на странице прогресса за неделю. Тап на "Начать следующую" → мини-апп сразу в режиме тренировки B.

**Важно:** вместо "4 из 4 тренировок" говорим про факт и мягко сравниваем с планом. План — ориентир, не KPI, давить не надо.

**Handoff:** **Бот → мини-апп.** Обе inline-кнопки ведут в мини-апп с разными контекстами.

**Edge cases:**
- Юзер пропустил всю неделю → "Сейчас не тренировался. Всё ок, жизнь. Если готов вернуться — напиши или жми кнопку [Начать следующую]".
- Юзер сделал много сверх плана (6 тренировок при плане 3) → "Мощно! Но не забудь про восстановление. Следующую можешь отложить на день".
- Первая неделя — отдельное поздравление ("Поздравляю с первой неделей!"), небольшой чеклист "что дальше".

---

### 3.3 🟡 Ежемесячный пост-анализ плана

**Триггер:** раз в 4 недели (или при завершении программы), node-cron в Railway.

**Цель:** сопоставить изначальный план с фактическими тренировками и предложить корректировку. Потому что юзер тренируется в своём темпе — нам важно показать, работает ли программа под его реальный ритм.

**История:**
Саша через месяц получает от бота:

> 🔄 **Месяц на программе Split A/B**
>
> *Планировалось:* 12 тренировок (3 в неделю).
> *Факт:* 10 тренировок. Средний интервал: 2.8 дня.
> *Распределение:* 6×A, 4×B — баланс A/B немного сдвинут к верху тела.
>
> *Прогресс:* тоннаж +12% к старту, новые рекорды в жиме и тяге.
>
> 💡 **Тренер замечает:**
> 1. Реальная частота 2.5/нед, не 3 — программа из 3 тренировок чуть перегружена.
> 2. Сдвиг к A означает, что спина/ноги отстают от плана.
>
> 🎯 **Варианты:**
> [Сохранить как есть] [Сократить до 2/нед] [Сбалансировать A/B] [Собрать новую программу]

Юзер выбирает вариант. Бот либо корректирует текущую программу, либо уходит в сценарий 2.7 (новая программа через диалог).

**Ключевые экраны:**
- Всё в боте. Если юзер выбрал "Сбалансировать" или "Сократить" — изменения применяются к Program в БД, в мини-аппе уже новая версия при следующем открытии.

**Handoff:** возможен переход в сценарий 2.7 (новая программа) или в мини-апп для просмотра обновлённой.

**Edge cases:**
- Юзер сделал меньше 3 тренировок за месяц → мягкий разбор "какой ритм реалистичен?" + предложение 1-2 тренировки в неделю.
- Юзер проигнорировал сообщение и не ответил → ничего не меняем, через 4 недели повторный анализ.
- Юзер сделал много (15+ тренировок при плане 12) → предупреждение про восстановление + предложение более агрессивной программы.

**Метрика успеха:** % юзеров, выбравших один из вариантов (engagement with the analysis); ретеншн после первого ежемесячного анализа (гипотеза: юзеры, которые увидели персональный разбор, лучше остаются в продукте).

---

### 3.4 🟡 Замена упражнения на лету

**Триггер:** в мини-аппе юзер хочет заменить упражнение (тренажёр занят / больно / не хочется).

**Цель:** за 3 тапа получить разумную замену.

**История:**
Саша подходит к скамье для жима — занято на 15 минут. Тапает на упражнение в экране тренировки → "⋯" → "Заменить". Получает 3 варианта от AI: "Жим гантелей лёжа (похоже, 90% точность)", "Отжимания с весом на поясе (80%)", "Жим в смит-машине (75%)". Выбирает первое — упражнение заменяется в текущей тренировке, автоподстановка пересчитывается на основе истории по гантелям.

**Ключевые экраны:**
- Меню упражнения (перейти к видео, заменить, пропустить, заметка)
- Модалка с 3 вариантами замены
- Возврат в экран тренировки уже с новым упражнением

---

### 3.5 🟡 История тренировок

**Триггер:** юзер хочет пересмотреть прошлые тренировки.

**Цель:** найти, сколько недавно жал, проверить прогресс в конкретном упражнении.

**История:**
Вика перед завтрашним жимом хочет посмотреть, сколько жала в прошлый раз. Открывает "История" → список прошлых тренировок с датами и основными упражнениями. Тап на тренировку → детально: все упражнения, все подходы. Или во вкладке "По упражнениям" → фильтрует "Жим лёжа" → видит график весов и последние подходы.

**Ключевые экраны:**
- Список тренировок (карточки с датой, длительностью, объёмом)
- Детали тренировки (все подходы)
- Разрез по упражнениям (фильтр + хронология)

---

### 3.6 🟡 Библиотека упражнений

**Триггер:** юзер хочет узнать про технику незнакомого упражнения (без фото), найти альтернативу или просто полистать.

**История:**
Саша слышал про "болгарский сплит-присед", но не знает, как делать. Открывает мини-апп → "Упражнения" → поиск "болгарский" → карточка с описанием техники, видео с YouTube, мышцы-мишени, альтернативы.

**Ключевые экраны:**
- Список (группы мышц / оборудование — фильтр)
- Карточка упражнения (видео, инструкция, типичные ошибки, альтернативы)

---

### 3.7 🟡 Редактирование программы

**Триггер:** юзер хочет подстроить программу под себя (убрать упражнение, добавить, поменять порядок).

**История:**
Вика видит, что в программе есть "жим в хаммере", а в её зале этого тренажёра нет. Идёт в "Программа" → тап на упражнение → "Убрать"/"Заменить" → обновление программы.

**Ключевые экраны:**
- Обзор программы (недельный календарь)
- Редактор дня (drag-drop упражнений, добавление, удаление)

## 4. Коммерческая версия (v2+, итерация 7 и далее)

Здесь кратко — детали прорабатываем позже, когда MVP пройдёт валидацию.

### 4.1 🔵 Триал и подписка

После онбординга — 7 или 14 дней полного доступа бесплатно. По окончании триала юзер видит paywall при попытке запустить тренировку / задать вопрос в чате / получить программу. Оплата через Telegram Stars или Stripe.

**Ключевые экраны:** paywall с фичами, выбор тарифа (мес/год), экран успешной оплаты, управление подпиской (отмена, изменение).

### 4.2 🔵 Экспорт / шаринг

Поделиться прогрессом / картой мышц / завершённой тренировкой в формате красивой картинки, которая отправляется в любой чат Telegram.

### 4.3 🔵 Друзья и челленджи

Добавление друзей через Telegram-контакт → видеть их тренировки (с их разрешения), общие челленджи (например, "30 дней с тренировками").

### 4.4 🔵 Кардио / домашние тренировки / йога

Отдельные типы активностей. Домашние — с видео. Кардио — трекинг времени/дистанции. Йога — последовательности поз.

### 4.5 🔵 Голосовой ввод в зале

Кнопка микрофона → "семьдесят пять на десять" → AI парсит → подход залогирован. Позволяет вообще не доставать телефон из кармана.

### 4.6 🔵 Видеоанализ техники

Юзер снимает видео подхода → AI оценивает технику → текстовая обратная связь. Требует продвинутой модели и хорошего освещения.

### 4.7 🔵 Интеграции

Apple Health / Google Fit: вес, пульс, сон. Whoop / Oura: recovery score → подстройка нагрузки на день.

### 4.8 🔵 Питание

Учёт БЖУ, рекомендации. Возможно — фото еды → распознавание.

## 5. Handoffs: бот ↔ мини-апп

Ключевые точки, где бот и мини-апп передают пользователя друг другу.

### 5.1 Бот → мини-апп

**Формат deep-link:** `startapp` параметр Telegram Mini App. Схема: `<тип>_<параметры...>`, разделитель `_`. Парсинг `const [type, ...params] = raw.split('_')`.

| Триггер в боте | `startapp` | Куда ведёт в мини-аппе |
|---|---|---|
| `/start` или открытие мини-аппа (онбординг пройден) | — (пусто) | Главный экран "Программа" |
| `/start` до онбординга | — (пусто, онбординг в боте) | Главный экран (но обычно юзер в этот момент не нажимает) |
| Inline "Открыть программу" после онбординга | `program` | Главный экран "Программа" |
| `/workout` или "Начать следующую" (в weekly-сводке) | `next` | Экран тренировки сразу в режиме логирования |
| Inline "Открыть в тренировке" (после фото тренажёра) | `exercise_<slug>` | Режим тренировки с уже выбранным упражнением |
| Inline "Посмотреть детали" (post-workout сводка) | `workout_<id>` | Детали конкретной тренировки (история) |
| Inline "Посмотреть отчёт" (weekly/monthly сводка) | `progress_week_<yyyyww>` или `progress_month_<yyyymm>` | Страница прогресса с выбранным периодом |
| Inline "Заменить на X" | `next` | Экран тренировки; backend уже обновил Program, фронт фетчит свежее |

Технически: используем [Telegram Mini App deep linking](https://core.telegram.org/bots/webapps#direct-link-mini-apps) через `startapp` параметр. На фронте делаем централизованный парсер `src/utils/deepLink.js`.

### 5.2 Мини-апп → бот

| Триггер в мини-аппе | Переход | Что передаём |
|---|---|---|
| Кнопка "Задать вопрос тренеру" в экране упражнения | Закрывает мини-апп, открывает чат с ботом | Через `Telegram.WebApp.sendData()` — бот получает контекст и сразу присылает фокусированное сообщение ("Ты сейчас на жиме лёжа. Спрашивай!") |
| "Сфотографировать тренажёр" | Закрывает мини-апп, просит отправить фото в чат | Бот ждёт следующего фото с привязкой к контексту тренировки |
| "Поделиться тренировкой" | Отправляет в любой чат Telegram | Через `Telegram.WebApp.switchInlineQuery` или share-API |

### 5.3 Общие правила

- **Контекст не теряем.** Если юзер посреди тренировки ушёл в бот — при возврате в мини-апп продолжаем с того же места.
- **Не делаем круговерти.** Не отправляем юзера "бот → мини-апп → бот → мини-апп" — максимум один handoff на шаг.
- **Явные "я в другом месте"-пометки.** Если юзер в мини-аппе, но что-то важное случилось в боте (например, бот ответил на вопрос) — показываем бейдж/плашку "1 новое сообщение в чате".

## 6. Навигация и главный экран мини-аппа

Нижняя таб-бар с 4 разделами (MVP):

1. **🏋️ Программа** (домашняя вкладка) — активная программа + карточка "Следующая тренировка" с кнопкой "Начать". Без привязки к дням недели.
2. **📊 Прогресс** — графики, карта мышц, рекорды, история тренировок.
3. **📚 Библиотека** — каталог упражнений с видео.
4. **⚙️ Профиль** — данные юзера, список программ (активная + история), выход.

Ключевые принципы:

- **"Программа" — самый быстрый путь к запуску тренировки.** Открыл мини-апп → тап на "Начать" → вводишь первый подход. Два тапа между Telegram и весом на штанге.
- **Нет "сегодняшней тренировки"** — юзер сам решает, когда и какую сделать. Мини-апп показывает "Следующая в программе = X", но не давит датой.
- **Главный экран отвечает на вопрос "что мне делать?"**, а не "во сколько сегодня тренировка?". Дисциплина тайминга — забота юзера, наша — забота о программе.
